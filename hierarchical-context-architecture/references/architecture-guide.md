## 1. 전략적 가치

컨텍스트 엔지니어링의 핵심은 더 많은 텍스트를 주는 것이 아니라, 더 높은 신호를 가진 정보를 올바른 시점에 주는 것이다.
대규모 코드베이스에서는 다음 문제가 반복된다.

- 전역 문서가 비대해져 실제로 필요한 규칙을 찾기 어려움
- 대화가 길어질수록 예전 결정과 예외 규칙이 희미해짐
- 서브시스템 세부사항이 전역 규칙을 오염시켜 다른 작업의 정확도를 떨어뜨림
- 세션 요약 또는 컴팩션 이후 중요한 결정이 사라짐

이 스킬은 이런 문제를 해결하기 위해 컨텍스트를 다음 3계층으로 나눈다.

- 전역 앵커: 루트 `CLAUDE.md`
- 서브시스템 지식: 각 디렉토리의 `CONTEXT.md`
- 범용 협업 규칙: 필요한 위치의 `AGENTS.md`

## 2. 핵심 설계 원칙

### 2.1 Locality

정보는 설명하는 코드와 최대한 가까운 곳에 둔다.
예를 들어 `src/api/`만의 인증 예외를 루트 문서에 넣지 않고 `src/api/CONTEXT.md`에 둔다.
이렇게 해야 다른 작업이 인증 규칙에 불필요하게 끌려가지 않는다.

### 2.2 Progressive Disclosure

모든 세부사항을 한 번에 주입하지 않는다.
먼저 유지할 것은 다음과 같은 가벼운 식별자다.

- 디렉토리 경로
- 핵심 파일 이름
- 요약 한 줄
- 하위 문서 링크

상세 로직, 긴 예시, 로그, 스키마는 실제로 필요할 때만 연다.

### 2.3 Deterministic Overrides

지침 충돌이 생기면 더 구체적이고 더 가까운 문서가 우선한다.
예를 들어 `src/CONTEXT.md`가 일반적인 네이밍 규칙을 정의하고, `src/api/CONTEXT.md`가 API 타입 규칙을 더 강하게 제한하면 `src/api/` 아래 작업에서는 후자가 우선한다.

### 2.4 Context as Runtime Infrastructure

컨텍스트 문서는 설명서가 아니라 실행 품질을 좌우하는 런타임 인프라다.
같은 문제가 반복되면 코드를 고치는 것과 동시에 관련 문서를 수정해야 한다.

## 3. 파일 계층 표준

### 3.1 `CLAUDE.md`

루트 `CLAUDE.md`는 다음만 남기는 것을 권장한다.

- 빌드·테스트·실행 명령
- 전역 아키텍처 결정
- 공통 환경 변수와 제약
- 조직 수준 금지사항 또는 안전 규칙
- 하위 `CONTEXT.md` 트리의 진입점 안내

루트 `CLAUDE.md`는 200라인 이내를 권장한다.
이 문서는 컴팩션 이후 다시 읽는 핵심 앵커 역할을 맡기 때문이다.

다음 내용은 루트 문서에서 빼는 편이 좋다.

- 자주 바뀌는 작업 상태
- 상세 API 문서
- 긴 스타일 가이드
- 대량 로그
- 특정 하위 디렉토리만 아는 예외 규칙

### 3.2 `CONTEXT.md`

`CONTEXT.md`는 서브시스템 전용 지식 트리의 노드다.
권장 항목은 다음과 같다.

- 이 디렉토리의 책임과 경계
- 핵심 파일과 관계
- 로컬 워크플로 또는 위험 요소
- 상위 규칙에서 달라지는 점
- 더 깊은 하위 컨텍스트 링크

좋은 `CONTEXT.md`는 구현 세부 코드를 길게 복사하지 않고, “어디를 보면 되는지”와 “왜 그렇게 되어 있는지”를 먼저 알려준다.

### 3.3 `AGENTS.md`

`AGENTS.md`는 도구 간 이식성을 고려한 범용 레이어다.
권장 항목은 다음과 같다.

- 협업 규칙
- 리뷰 방식
- 역할 분리 규칙
- 공통 출력 계약
- 저장소 전반에서 지켜야 할 간단한 마크다운 규칙

다음은 피한다.

- IDE 전용 메타데이터
- 특정 도구에서만 의미 있는 장문의 설정
- 자주 바뀌는 로컬 임시 상태

## 4. 발견·로드 메커니즘

작업 대상 파일에서 시작해 상위 디렉토리로 올라가며 문서를 읽는 양파형 구조를 기본으로 한다.

예시: `src/api/auth.ts`를 수정할 때

1. `/CLAUDE.md`
2. `/src/CONTEXT.md`
3. `/src/api/CONTEXT.md`
4. `src/api/auth.ts`의 실제 코드와 최신 변경 사항

이 순서의 목적은 다음과 같다.

- 먼저 전역 제약을 고정하기
- 그 다음 서브시스템 공통 패턴을 불러오기
- 마지막에 가장 구체적인 예외와 현재 작업 상태를 얹기

### Override Rule 메모

- 더 하위 문서가 더 구체적인 규칙을 가진다.
- 하위 문서가 상위 규칙을 바꾸면 “무엇을 왜 덮어쓰는지”를 적는다.
- 상위 문서에는 예외 자체보다 예외가 존재한다는 사실만 남겨도 된다.

## 5. 토큰 효율 최적화

### 5.1 Anchored Summaries

긴 대화가 누적되면 모든 로그를 그대로 유지하지 말고 다음만 보존한다.

- 확정된 아키텍처 결정
- 미해결 이슈의 현재 상태
- 다음 행동에 필요한 근거

반대로 대량 출력은 요약하거나 별도 파일로 분리한다.

### 5.2 Instruction/Data Separation

규칙과 데이터를 한 문단에 섞지 않는다.
다음처럼 역할을 분리해 두면 지침 누출을 줄이기 쉽다.

```xml
<instructions>
  - 루트 문서는 200라인 이내로 유지한다.
  - API 타입은 @src/api/types.ts를 우선 참조한다.
</instructions>

<context>
  현재 작업 경로: src/api/
  상위 컨텍스트: ../../CONTEXT.md
</context>

<data>
  BUILD_LOG: Error at line 45...
</data>
```

XML이 필수는 아니지만, 규칙과 로그의 경계를 분명히 드러내는 구조를 유지하는 것이 중요하다.

### 5.3 Prefix Preservation

정적 지침은 프롬프트 앞부분에, 동적 데이터는 뒤에 둔다.
이렇게 해야 캐시 재사용성과 규칙 회상률을 함께 지키기 쉽다.

## 6. 운영 지침과 검증 루프

### 6.1 Fix the Rules

AI가 같은 종류의 실수를 반복하면 다음 둘 다 수행한다.

1. 코드 수정
2. 실수를 허용한 컨텍스트 문서 수정

이 루프를 닫지 않으면 같은 실패가 다른 세션에서 다시 나타난다.

### 6.2 3단계 검증 태스크

#### 1) Reference Integrity

- 링크된 `CONTEXT.md`, `AGENTS.md`, `CLAUDE.md`가 실제로 존재하는가?
- 상대 경로가 깨지지 않았는가?
- 고립된 `CONTEXT.md` 후보가 있는가?

#### 2) Code Reference Validation

- 문서에 적힌 파일 경로가 실제 코드 구조와 맞는가?
- 오래된 모듈명이나 이동된 경로를 아직 문서가 가리키고 있지 않은가?

#### 3) Content Accuracy

- 문서의 기술적 주장과 실제 구현 패턴이 일치하는가?
- 더 이상 존재하지 않는 아키텍처 결정을 계속 설명하고 있지 않은가?
- 로컬 예외가 전역 규칙으로 잘못 승격되어 있지 않은가?

자동화는 1, 2단계를 우선 커버하고, 3단계는 주기적인 수동 리뷰를 추가하는 방식이 현실적이다.

## 7. Quick Start 체크리스트

- 루트에 200라인 이내 `CLAUDE.md`를 만든다.
- 주요 서브시스템마다 `CONTEXT.md`를 둔다.
- 범용 협업 규칙을 `AGENTS.md`에 둔다.
- 하위 컨텍스트는 링크로 서로 연결한다.
- 긴 로그와 대형 예시는 규칙 문서에서 분리한다.
- 변경 후 자동 검증을 실행한다.
- AI가 실수한 유형은 문서 규칙으로 역전파한다.
